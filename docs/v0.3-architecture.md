# Workshop v0.3 Architecture — Channel Autonomy

> Replicate Kagura's Discord multi-channel agent experience natively in Workshop.

## Context

Workshop v0.2 has multi-agent chat with rooms, @mention routing, and SQLite persistence. But rooms are passive containers — humans must type to trigger agent activity. In Kagura's Discord setup, channels are **autonomous**: cron-driven loops read channel files and TODO sections, agents self-activate, and a heartbeat keeps the system alive. v0.3 bridges this gap.

### Discord Experience → Workshop Feature Map

| Discord Experience | Workshop v0.3 Feature | Section |
|---|---|---|
| Channel files (`channels/<name>.md`) with positioning, guidelines, north star | Channel metadata system | §1 |
| Single `TODO.md` with per-channel sections, pin sync | Global TODO + per-channel task sections | §2 |
| Per-channel cron loops (work-loop, study-loop, etc.) | Cron-driven channel autonomy | §3 |
| Channel-patrol (巡检) | Automated patrol → control room summary | §4 |
| File-system-based cross-channel coordination | Cross-channel notification events | §5 |
| Luna types in channel, agent sees on next cron | Real-time human intervention | §6 |
| TODO items: exists or deleted | Task lifecycle with states | §7 |

## §1. Channel System — Channel Files & Metadata

**Discord equivalent:** Each channel has `channels/<name>.md` containing positioning (what the channel does), guidelines (cron behavior rules), and north star (channel-level goals).

### Data Model

```typescript

  name: string;
  agents: string[];
  agentConfigs?: { id: string; requireMention: boolean }[];

  status: 'active' | 'completed' | 'archived';

  positioning: string;                     // what this channel does
  guidelines: string;                      // rules for cron-triggered behavior
```

- **Channel header**: shows channel type badge (`daily` / `project` / `meta`) and a one-line positioning summary below the channel name.

## §2. North Star / TODO Integration

**Discord equivalent:** Single `TODO.md` with per-channel sections. Pin sync keeps `#kagura-dm` pins up to date. North star (`wiki/strategy.md`) is pinned globally.

### Data Model

```typescript
interface TodoItem {
  id: string;
  section: string;           // e.g. "🔨 打工", "📚 学习" — maps to channel.todoSection
  content: string;           // the task text
  status: 'pending' | 'in_progress' | 'review' | 'done';
  assignedChannel: string | null;  // channel ID that owns this item
  createdAt: string;
  updatedAt: string;
  staleDays: number;         // auto-computed: days since last update
}

interface NorthStar {
  id: string;
  scope: 'global' | string;  // 'global' or channel ID
  content: string;            // markdown content
  updatedAt: string;
}

interface Pin {
  id: string;
  channelId: string;
  type: 'todo_section' | 'north_star' | 'custom';
  sourceId: string;           // todo section name or north star ID
  content: string;            // rendered snapshot
  updatedAt: string;
}
```

### SQLite Schema

```sql

  content TEXT NOT NULL,
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
```

### API Additions

```typescript

| { type: 'todo_create'; section: string; content: string }
| { type: 'todo_update'; itemId: string; status?: string; content?: string }
| { type: 'todo_delete'; itemId: string }
| { type: 'north_star_get'; scope?: string }
| { type: 'north_star_set'; scope: string; content: string }
| { type: 'pin_list'; channelId: string }

// Server → Client
| { type: 'todo_items'; items: TodoItem[] }
| { type: 'todo_changed'; item: TodoItem; action: 'created' | 'updated' | 'deleted' }
| { type: 'north_star'; star: NorthStar }
| { type: 'pin_list'; channelId: string; pins: Pin[] }
| { type: 'pin_updated'; channelId: string; pin: Pin }
```

### Pin Sync Logic

When a TODO item changes or north star is updated:
1. Find all pins referencing the changed source.
2. Re-render the pin content (snapshot of the current TODO section or north star text).
3. Broadcast `pin_updated` to all connected clients.
4. If a channel's `todoSection` matches the changed section, auto-create a pin in that channel if none exists.

### UI Changes

- **Pinned items bar**: horizontal bar below the channel header showing pinned TODO sections and north star. Clicking a pin expands it.
- **TODO panel**: global TODO view accessible from sidebar (like Discord's pinned messages view). Sections are collapsible. Each item shows status badge.
- **Stale indicator**: items unchanged for 3+ days get a warning badge (maps to Discord's "拖 3 天必须做或删" rule).

### Priority: **P0** — Drives the cron behavior in §3 and task lifecycle in §7.

## §3. Cron-Driven Channel Autonomy

**Discord equivalent:** Each channel has its own cron (work-loop every hour, study-loop every 30m, etc.). Cron fires → agent reads channel file + TODO section → executes → updates TODO.

### Design

Workshop server runs an internal cron scheduler. For each channel with `cronEnabled: true` and a `cronSchedule`, the scheduler:

1. Fires at the scheduled time.
2. Assembles a **cron prompt** from the channel's metadata:
   ```
   [Channel: #{channel.name}]
   [Type: {channel.type}]

   ## Positioning
   {channel.positioning}

   ## Guidelines
   {channel.guidelines}

   ## North Star
   {channel.northStar}

   ## Current TODO ({channel.todoSection})
   {rendered TODO items for this section, with status}

   ## Recent Activity
   {last 10 messages in this channel}

   ---
   Execute according to guidelines. Update TODO items as you work.
   ```
3. Sends the prompt via `chat.send` to the channel's agents (respecting `requireMention` rules — non-mention agents get the cron message, mention-only agents don't).
4. Agent response flows through normal message pipeline → stored in SQLite → broadcast to frontend.

### Data Model Additions

```typescript
interface CronExecution {
  id: string;
  channelId: string;
  scheduledAt: string;       // when it was supposed to fire
  firedAt: string;           // when it actually fired
  promptSnippet: string;     // first 200 chars of assembled prompt (for debugging)
  status: 'fired' | 'completed' | 'error';
  completedAt: string | null;
}
```

### SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS cron_executions (
  id TEXT PRIMARY KEY,
  channel_id TEXT NOT NULL,
  scheduled_at TEXT NOT NULL,
  fired_at TEXT NOT NULL DEFAULT (datetime('now')),
  prompt_snippet TEXT,
  status TEXT NOT NULL DEFAULT 'fired',
  completed_at TEXT,
  FOREIGN KEY (channel_id) REFERENCES channels(id)
);
```

## §4. Channel Patrol

**Discord equivalent:** Envisioned but not yet implemented in Discord. Periodic inspection of all channels, summarizing activity to a control room (`#kagura-dm`).

### Design

Channel patrol is a **special cron job** on the meta/control channel (e.g., `#kagura-dm`). Instead of executing tasks, it:

1. Reads the last N messages from every active channel since the last patrol.
2. Reads current TODO status across all sections.
3. Assembles a patrol prompt:
   ```
   [Channel Patrol — #{controlChannel.name}]

   Summarize the status of all channels since last patrol ({lastPatrolTime}).

   ## Channel Activity
   {for each active channel:}
   ### #{channel.name} ({channel.type})
   - Messages since last patrol: {count}
   - Last activity: {timestamp}
   - Active TODO items: {count pending}/{count in_progress}/{count done}
   - Summary of recent messages: {last 3 messages, truncated}

   ## Global TODO Status
   {sections with item counts by status}

   ---
   Provide a concise status update. Flag anything that looks stuck, overdue, or needs attention.
   ```
4. Agent processes and responds with a summary in the control channel.

### Data Model

```typescript
interface PatrolConfig {
  controlChannelId: string;   // which channel receives patrol summaries
  schedule: string;           // cron expression (e.g. "0 */3 * * *" = every 3 hours)
  enabled: boolean;
  lastPatrolAt: string | null;
  channelFilter: string[];    // channel IDs to include (empty = all active)
}
```

### SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS patrol_config (
  id TEXT PRIMARY KEY DEFAULT 'default',
  control_channel_id TEXT NOT NULL,
  schedule TEXT NOT NULL DEFAULT '0 */3 * * *',
  enabled INTEGER NOT NULL DEFAULT 0,
  last_patrol_at TEXT,
  channel_filter TEXT NOT NULL DEFAULT '[]',   -- JSON array of channel IDs
  FOREIGN KEY (control_channel_id) REFERENCES channels(id)
);
```

### API Additions

```typescript
// Client → Server
| { type: 'patrol_config_get' }
| { type: 'patrol_config_set'; config: Partial<PatrolConfig> }
| { type: 'patrol_trigger' }                    // manual patrol now

// Server → Client
| { type: 'patrol_config'; config: PatrolConfig }
| { type: 'patrol_fired'; controlChannelId: string }
```

### UI Changes

- **Patrol settings**: accessible from the control channel's settings panel. Configure schedule, enable/disable, select which channels to include.
- **Patrol badge**: control channel gets a shield icon in the sidebar to distinguish it.
- **Manual patrol button**: "Patrol Now" button in control channel header.

### Priority: **P1** — Useful but not blocking. The human can manually check channels. Implement after §1-§3 are stable.

## §5. Cross-Channel Notification

**Discord equivalent:** Currently channels communicate only through shared `TODO.md` file. No direct channel-to-channel messaging. This is identified as Discord Gap #1.

### Design

When something notable happens in one channel, Workshop can notify other channels. Two mechanisms:

#### 5a. Automatic TODO-driven notifications

When an agent updates a TODO item's status (e.g., `in_progress` → `done`), Workshop checks if any other channel references the same TODO section or has a dependency. If so, it posts a system notification in the target channel.

#### 5b. Agent-initiated cross-post

An agent can request posting to another channel by using a structured format in its response:

```
@notify #channel-name: message content here
```

Workshop server parses this pattern, posts the notification to the target channel as a system message, and optionally triggers that channel's agents.

### Data Model

```typescript
interface Notification {
  id: string;
  sourceChannelId: string;
  targetChannelId: string;
  content: string;
  trigger: 'todo_change' | 'agent_crosspost' | 'patrol';
  todoItemId: string | null;      // if triggered by TODO change
  createdAt: string;
  read: boolean;
}
```

### SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS notifications (
  id TEXT PRIMARY KEY,
  source_channel_id TEXT NOT NULL,
  target_channel_id TEXT NOT NULL,
  content TEXT NOT NULL,
  trigger TEXT NOT NULL,
  todo_item_id TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  read INTEGER NOT NULL DEFAULT 0,
  FOREIGN KEY (source_channel_id) REFERENCES channels(id),
  FOREIGN KEY (target_channel_id) REFERENCES channels(id)
);
```

### API Additions

```typescript
// Server → Client (pushed automatically)
| { type: 'notification'; notification: Notification }
| { type: 'notification_badge'; channelId: string; unreadCount: number }

// Client → Server
| { type: 'notification_mark_read'; channelId: string }
```

### Cross-Post Parsing (in Router)

```typescript
// In router.ts, after receiving an agent message:
const crossPostPattern = /@notify\s+#([\w-]+):\s*(.+)/s;
const match = content.match(crossPostPattern);
if (match) {
  const targetChannelId = match[1];
  const notifyContent = match[2];
  this.postNotification(sourceChannelId, targetChannelId, notifyContent, 'agent_crosspost');
}
```

### UI Changes

- **Unread notification badge**: red dot with count on channel names in sidebar (like Discord's unread indicator).
- **Notification message style**: system messages from other channels rendered with a special "cross-post" style — shows source channel name and a link to jump there.
- **Notification settings per channel**: opt-in/out of receiving notifications from specific channels.

### Priority: **P1** — Important for multi-channel coordination but can be approximated by patrol (§4) initially.

## §6. Real-Time Human Intervention

**Discord equivalent:** Luna types in a channel, agent sees it on next cron trigger. Gap: not real-time. Workshop can do better.

### Design

Workshop already handles real-time human messages — `send_message` immediately forwards to agents. The gap is that in cron-driven mode, the agent's session context was set by the last cron prompt, not the human's message. Two approaches:

#### 6a. Immediate forwarding (already works)

Human types → `chat.send` to agent → agent responds. This is v0.2 behavior. Works for `requireMention: false` agents.

#### 6b. Cron context injection

When a human posts in a cron-driven channel between cron fires, the message is stored normally. On the **next cron fire**, the assembled prompt includes these human messages in the "Recent Activity" section, so the agent sees them in context.

#### 6c. Urgent interrupt (new)

For time-sensitive intervention, add an `@urgent` prefix. When Workshop sees `@urgent`, it:
1. Sends the message to the agent immediately (normal flow).
2. Prepends the channel's guidelines to the message so the agent has context even outside cron.
3. Resets the cron timer (so the next cron doesn't re-trigger too soon after the interrupt).

### Data Model

No new tables needed. Add a column to messages:

```sql
ALTER TABLE messages ADD COLUMN is_urgent INTEGER NOT NULL DEFAULT 0;
```

### API Changes

The `send_message` handler checks for `@urgent` prefix. No new message types needed — it's parsed from content.

### UI Changes

- **Urgent styling**: messages with `@urgent` get a red-bordered highlight.
- **Channel activity indicator**: in the sidebar, channels with unread human messages (that agents haven't responded to) show a yellow dot. Channels with pending cron get a clock icon.
- **Intervention status**: in the channel header, show "Agent last active: {time}" so the human knows if the agent is responsive.

### Priority: **P1** — v0.2 already has basic real-time. The urgent interrupt and cron-context injection are refinements.

## §7. Task Lifecycle

**Discord equivalent:** TODO items only have two states: exists or deleted. Gap #6.

### Design

Extend TODO items with a proper lifecycle:

```
pending → in_progress → review → done
                  ↓         ↓
               pending   in_progress  (rejection/rework)
```

### State Transitions

| From | To | Trigger |
|---|---|---|
| `pending` | `in_progress` | Agent picks up task during cron, or human assigns |
| `in_progress` | `review` | Agent reports task complete, awaiting human check |
| `in_progress` | `pending` | Agent encounters blocker, defers task |
| `review` | `done` | Human approves, or auto-approve after N hours |
| `review` | `in_progress` | Human requests changes |
| `done` | *(archived)* | Removed from active TODO, kept in history |

### Stale Detection

- **3-day rule**: items `pending` for 3+ days are flagged. Patrol (§4) and cron prompts (§3) call these out.
- **In-progress timeout**: items `in_progress` for 24+ hours without activity are flagged as potentially stuck.

### Data Model

Already defined in §2's `TodoItem` interface. The `status` field covers the lifecycle:

```typescript
status: 'pending' | 'in_progress' | 'review' | 'done';
```

Additional tracking:

```sql
-- Add to todo_items table
ALTER TABLE todo_items ADD COLUMN assigned_agent TEXT;       -- which agent is working on it
ALTER TABLE todo_items ADD COLUMN stale_days INTEGER NOT NULL DEFAULT 0;  -- computed on read

-- History log for audit
CREATE TABLE IF NOT EXISTS todo_history (
  id TEXT PRIMARY KEY,
  todo_item_id TEXT NOT NULL,
  from_status TEXT,
  to_status TEXT NOT NULL,
  changed_by TEXT NOT NULL,      -- agent ID or 'user'
  changed_at TEXT NOT NULL DEFAULT (datetime('now')),
  note TEXT,
  FOREIGN KEY (todo_item_id) REFERENCES todo_items(id)
);
```

### Agent Integration

Agents can update TODO status by including structured markers in their responses:

```
@todo:start <item-id>       → pending → in_progress
@todo:review <item-id>      → in_progress → review
@todo:done <item-id>        → review → done (or in_progress → done for auto-approve items)
@todo:block <item-id> <reason>  → in_progress → pending
```

Workshop server parses these markers, updates the database, triggers pin sync (§2) and notifications (§5).

### UI Changes

- **Status badges**: colored pill next to each TODO item — gray (pending), blue (in progress), yellow (review), green (done).
- **Kanban view**: optional toggle in the TODO panel to show items in a kanban board layout (columns = states).
- **Transition buttons**: human can manually change an item's status by clicking the status badge.
- **History tooltip**: hover over a TODO item to see its status transition history.

### Priority: **P0** — Core to the cron execution model. Agents need states to report progress.

## Implementation Phases

### Phase 1: Foundation (P0)

**Goal:** Channels become autonomous units with metadata and cron.

1. **§1 Channel metadata** — Add columns, API, settings panel.
2. **§7 Task lifecycle** — `todo_items` table, status transitions, agent markers.
3. **§2 TODO/NorthStar** — `todo_items`, `north_stars`, `pins` tables, pin sync logic.
4. **§3 Cron scheduler** — `ChannelCronManager`, cron prompt assembly, execution logging.

**Estimated scope:** ~800 LoC server, ~600 LoC frontend.

### Phase 2: Coordination (P1)

**Goal:** Channels communicate and the human has full visibility.

5. **§4 Channel patrol** — Patrol config, patrol prompt assembly, control channel.
6. **§5 Cross-channel notifications** — `@notify` parsing, notification badges, TODO-change triggers.
7. **§6 Urgent intervention** — `@urgent` parsing, cron timer reset, context injection.

**Estimated scope:** ~500 LoC server, ~400 LoC frontend.

### Phase 3: Polish (P2)

8. **Kanban view** for TODO items.
9. **Cron execution dashboard** — visualize all channel cron timelines.
10. **Pin sync expansion** — any file/content can be pinned to any channel.
11. **Agent-to-agent messaging** — agents in one channel can directly message agents in another.

## Non-Goals (v0.3)

- **Multi-user auth** — single user (Kagura/Luna), no login.
- **Agent-to-agent direct messaging** — defer to v0.4. Cross-channel notification (§5) covers the main use case.
- **Mobile/responsive UI** — desktop browser only.
- **Streaming deltas** — still show final messages only. Streaming is a v0.4 polish item.
- **External integrations** — no Slack/Feishu bridging yet.

## Open Questions

- [ ] Should cron prompts go through the same session as human messages, or use a separate session key (e.g., `agent:kagura:cron:work`)? Same session means the agent sees conversation history. Separate session means a clean context each time.
- [ ] Should TODO items live only in SQLite, or also sync to a `TODO.md` file for backward compatibility with Discord workflows?
- [ ] How to handle agent responses that contain both a normal reply and `@todo:` / `@notify` markers — strip the markers from the displayed message?
- [ ] Should patrol run as a dedicated agent, or reuse an existing agent (e.g., Kagura) in the control channel?
